## Executive Summary

SafeRoute AI is an intelligent navigation application that empowers riders to collaboratively maintain road safety by reporting hazards and receiving AI-validated route recommendations. The prototype demonstrates a full-featured user experience with authentication, hazard reporting, smart route planning, and community-driven risk assessment.

### Core Value Proposition
- **For Riders**: Get safe route recommendations by avoiding hazard-prone areas
- **For Community**: Contribute road safety data through crowdsourced hazard reports
- **For System**: Build real-time risk intelligence through AI-validated submissions

## Key Design Decisions

### 1. Mock Data for UI Fidelity
- Used JSON hardcoding instead of real API to demonstrate UI without backend
- Easy swap later: Just replace mock_data.dart with API calls
- Maintains type safety through HazardReport model

### 2. SetState-Based Architecture
- Simple state management for prototype
- Can be upgraded to Provider/BLoC/Riverpod later
- Clear widget hierarchy and data flow

### 3. Material Design 3
- Modern, accessible aesthetics
- Built-in dark mode support
- Smooth transitions and animations
- Color system: Primary Green (safety), Risk Red (dangers)

### 4. Route-Based Navigation
- Named routes for common flows (auth, home)
- Dialog/modal navigation for secondary screens
- Clean separation of concerns

### 5. Map Mock Implementation
- Uses CustomPainter instead of real map SDK
- Demonstrates layout and interaction patterns
- Ready to integrate flutter_map for real maps

### 6. Throttled Async Operations
- Artificial delays mimic network calls (900ms-1200ms)
- Shows loading states and user feedback
- Easy to replace with real API calls

1. Update `_hazards` list in `report_hazard_screen.dart`

3. Update mock data entries in `mock_data.dart`

## Code Quality Standards

### Applied Styling
- Material Design 3 components exclusively
- Consistent padding (18, 14, 12 standardized)
- Widget composition over inheritance
- Clear separation of form logic and UI
- Proper resource cleanup (dispose methods)

### Naming Conventions
- Screens: `*_screen.dart` (snake_case file, PascalCase class)
- Widgets: `*_widget.dart` or `animated_*.dart`
- Models: `*_model.dart` or simple nouns (hazard_report.dart)
- Utility: `*_utils.dart` or `app_*.dart`
- Methods: camelCase, verb-first (_submitReport, _loadData)
- Private: underscore prefix for internal members

### Documentation Approach
- Self-documenting code preferred
- Inline comments for complex logic
- Parameter descriptions in constructor comments
- TODO markers for known limitations

## Troubleshooting Reference

### Common Issues
1. **Map dropdown crashes on value selection**
   → Ensure selected value exists in filtered list. Keep it in options or auto-adjust paired field.

2. **Theme doesn't persist after app restart**
   → Expected – theme toggle is session-scoped. Integrate SharedPreferences for persistence.

3. **Mock data not rendering**
   → Check hazardsJson format matches HazardReport.fromJson() expectations.

4. **Navigation state issues**
   → Verify route names match named route definitions in main.dart.
